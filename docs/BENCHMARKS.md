# Benchmarks: jsq vs jq vs jaq

`jsq`, [`jq`](https://jqlang.github.io/jq/), and [`jaq`](https://github.com/01mf02/jaq)
solve overlapping problems, so the fair question is: on the same file and the same query,
how do they compare on **wall time** and **memory**? `jaq` is a jq reimplementation in
Rust, which makes the trio especially informative — jq vs jaq isolates *C vs Rust*, while
jsq vs both isolates *parse-everything vs streaming*.

The short answer: jq and jaq both parse the whole document into the heap, so their memory
scales with file size. jaq is generally faster than jq (same approach, tighter
implementation), but it carries the same memory cost. `jsq` memory-maps the file and
streams a single pass, so its actual RAM stays nearly flat — tens of MB even on a 1 GB
file — and it pulls ahead on time as files grow. **The memory win is about the approach,
not the language: jsq beats jaq on RAM by the same wide margin it beats jq.**

The harness, generator, and queries are in [`../bench`](../bench) — everything here is
reproducible.

## Setup

- **Machine:** Apple M2 Max, 32 GB RAM, macOS (Darwin 25.2), SSD.
- **Tools:** `jsq` 1.0.0 (release build), `jq` 1.7.1, `jaq` 3.0.0, `hyperfine` 1.20.
- **Data:** a single JSON object `{"events":[…], "users":[…], "regions":[…]}` — one large
  fact array of order events (scalar fields plus a small nested `items` array), a 10k-row
  `users` dimension keyed by `user_id`, and a 5-row `regions` dimension keyed by region
  code. Only `events` grows with the target size; the two dimensions are byte-identical
  across every file so the join queries have a stable thing to join against. Generated by
  [`bench/gen.py`](../bench/gen.py), seeded for reproducibility.
- **Time:** `hyperfine` with 1 warmup run, reporting the **median** of 5 runs (3 at
  1 GB). Output is redirected to `/dev/null` so streaming the result doesn't skew timing.

### Two memory numbers

Both come from `/usr/bin/time -l` on a separate single run:

- **RAM** = *peak memory footprint* (`phys_footprint`). This is the dirty + compressed
  memory the process actually owns — the number **Activity Monitor shows in its
  "Memory" column**. It excludes clean, file-backed pages the kernel can drop for free.
- **RSS** = *maximum resident set size*. This also counts the memory-mapped source and
  index pages faulted in during the scan. Those pages are clean and reclaimable (the
  kernel can evict them without swapping), so RSS overstates the memory jsq truly needs.

For `jq` and `jaq` the two are nearly identical — it's all heap. For `jsq` they diverge
sharply, and **RAM is the honest "how much memory does this need" figure.** RSS is
included for completeness.

## Queries

Nine queries of increasing complexity, spanning filter/project, counts, single- and
multi-metric group-bys, single and chained joins, array `unnest`, a post-aggregate
`having` filter, and a combined join + unnest + filter + multi-metric "kitchen sink".

`jq` and `jaq` run the **same** filter (jaq is jq-compatible). The jq form is the
efficient idiomatic shape — a `reduce` into an object for every group-by, never the
memory-hungry `group_by`, and a hand-built index (`reduce … as $row ({}; …)`) for every
join — so neither jq nor jaq is strawmanned. **All three tools produce identical results,
verified before timing** (compared as canonical JSON; the only textual differences are
cosmetic — jsq emits NDJSON with spaces, and jq 1.7 preserves source literals like
`573.0` where jsq renders `573` for the same number).

The full text of every query follows. jsq queries are in [`bench/queries/qN.jsq`](../bench/queries);
the shared jq/jaq filter is in `qN.jq`.

### q1 — filter + project

Keep events over a threshold, project two fields. The one query that emits a row per
surviving event (millions of them), not a reduced summary.

```
# q1.jsq
from .events[] as e
where e.amount > 500
select { id: e.id, amount: e.amount }
```
```jq
# q1.jq
.events[] | select(.amount > 500) | {id, amount}
```

### q2 — filter + count

Count the events matching a predicate. Collapses the whole stream to one scalar.

```
# q2.jsq
from .events[] as e
where e.status == "paid"
aggregate { n: count() }
```
```jq
# q2.jq
reduce (.events[] | select(.status == "paid")) as $e (0; . + 1)
```

### q3 — group by, single sum

The canonical group-and-sum: total revenue per region.

```
# q3.jsq
from .events[] as e
aggregate { revenue: sum(e.amount) } by e.region
```
```jq
# q3.jq
reduce .events[] as $e ({}; .[$e.region] += $e.amount)
| to_entries | map({region: .key, revenue: .value}) | sort_by(.region) | .[]
```

### q4 — group by, three metrics, ordered

Sum, count, and average per region, sorted by revenue. Forces the parse-everything tools
to hold an intermediate object per group with multiple accumulators.

```
# q4.jsq
from .events[] as e
aggregate { revenue: sum(e.amount), n: count(), avg: avg(e.amount) } by e.region
order by .revenue desc
```
```jq
# q4.jq
reduce .events[] as $e ({};
  .[$e.region] |= ((. // {revenue:0, n:0})
    | {revenue: (.revenue + $e.amount), n: (.n + 1)}))
| to_entries
| map({region: .key, revenue: .value.revenue, n: .value.n, avg: (.value.revenue / .value.n)})
| sort_by(-.revenue) | .[]
```

### q5 — inner join + group by

Join each event to its user (10k-row dimension), then sum revenue by the user's tier.
jsq builds a memory-mapped foreign-key index for the join; the jq form builds the
equivalent index by hand with a `reduce`.

```
# q5.jsq
from .events[] as e
join .users[] as u on u.user_id == e.user_id
aggregate { revenue: sum(e.amount) } by u.tier
order by .tier
```
```jq
# q5.jq
(reduce .users[] as $u ({}; .[$u.user_id|tostring] = $u.tier)) as $tier
| reduce .events[] as $e ({}; .[$tier[$e.user_id|tostring]] += $e.amount)
| to_entries | map({tier: .key, revenue: .value}) | sort_by(.tier) | .[]
```

### q6 — chained (two-hop) join + group by

event → user → region: join events to users, then users to the regions dimension, and
sum revenue by the region's country name. Two index lookups per event.

```
# q6.jsq
from .events[] as e
join .users[] as u on u.user_id == e.user_id
join .regions[] as r on r.code == u.region
aggregate { revenue: sum(e.amount) } by r.country
order by .country
```
```jq
# q6.jq
(reduce .regions[] as $r ({}; .[$r.code] = $r.country)) as $country
| (reduce .users[] as $u ({}; .[$u.user_id|tostring] = $u.region)) as $ureg
| reduce .events[] as $e ({}; .[$country[$ureg[$e.user_id|tostring]]] += $e.amount)
| to_entries | map({country: .key, revenue: .value}) | sort_by(.country) | .[]
```

### q7 — unnest + group by

Flatten each event's nested `items` array (one row per line item) and sum quantity by
SKU. Tests array fan-out: the row count is larger than the event count.

```
# q7.jsq
from .events[] as e
unnest e.items as it
aggregate { qty: sum(it.qty) } by it.sku
order by .sku
```
```jq
# q7.jq
reduce (.events[] | .items[]) as $it ({}; .[$it.sku] += $it.qty)
| to_entries | map({sku: .key, qty: .value}) | sort_by(.sku) | .[]
```

### q8 — kitchen sink: join + unnest + filter + multi-metric + order

Everything at once: join to users, flatten `items`, filter to paid events, then per tier
count the line items and sum their quantity, ordered by quantity. The heaviest pipeline.

```
# q8.jsq
from .events[] as e
join .users[] as u on u.user_id == e.user_id
unnest e.items as it
where e.status == "paid"
aggregate { lines: count(), qty: sum(it.qty) } by u.tier
order by .qty desc
```
```jq
# q8.jq
(reduce .users[] as $u ({}; .[$u.user_id|tostring] = $u.tier)) as $tier
| reduce (.events[] | select(.status == "paid")) as $e ({};
    reduce $e.items[] as $it (.;
      .[$tier[$e.user_id|tostring]] |= ((. // {lines:0, qty:0})
        | .lines += 1 | .qty += $it.qty)))
| to_entries
| map({tier: .key, lines: .value.lines, qty: .value.qty})
| sort_by(-.qty) | .[]
```

### q9 — group by + having (post-aggregate filter)

Sum revenue and count per region over paid events, then keep only the groups whose
revenue clears a threshold. Tests filtering *after* aggregation, which jq must express as
a `map(select(…))` over the materialized groups.

```
# q9.jsq
from .events[] as e
where e.status == "paid"
aggregate { revenue: sum(e.amount), n: count() } by e.region
having .revenue > 400000
order by .revenue desc
```
```jq
# q9.jq
reduce (.events[] | select(.status == "paid")) as $e ({};
  .[$e.region] |= ((. // {revenue:0, n:0})
    | {revenue: (.revenue + $e.amount), n: (.n + 1)}))
| to_entries
| map({region: .key, revenue: .value.revenue, n: .value.n})
| map(select(.revenue > 400000))
| sort_by(-.revenue) | .[]
```

## Takeaways

- **Memory is the headline, and it's an architecture story.** jq and jaq both hold the
  entire parsed document on the heap — roughly **16× the file size** — so a 1 GB file
  needs many GB of RAM whichever you pick. `jsq` keeps its records in a memory-mapped
  index, not the heap, so its actual RAM (the Activity-Monitor figure) stays in the tens
  of MB regardless of file size. At 100 MB that's ~33 MiB vs ~1.5–1.7 GB — and rewriting
  jq in Rust (jaq) barely moves that number, because the cost is the *approach*, not the
  language.
- **jaq is the speed control, not a memory control.** jaq is usually faster than jq
  (often ~1.5–2× on the aggregates) — a fair measure of what a tight Rust implementation
  of the parse-everything model buys you. It does not close the memory gap with jsq at
  all, which is the cleanest evidence that streaming, not Rust, is what makes jsq frugal.
- **Joins and unnest don't change the picture.** The single join (q5), the two-hop chained
  join (q6), the array `unnest` (q7), and the combined kitchen-sink pipeline (q8) all keep
  jsq's owned memory flat: jsq builds a memory-mapped foreign-key index for joins rather
  than materializing the joined rows, so even a two-hop join over 7M events stays in the
  tens of MB while jq/jaq still hold the whole document plus their hand-built index.
- **mmap is why RSS and RAM differ for jsq.** jsq's larger RSS is mostly clean,
  file-backed pages (the source and the index) that the kernel can reclaim without
  swapping. That's also what lets jsq query files **larger than RAM** — something the
  parse-everything approach can't do at all.
- **Group-bys, joins, and `having` (q3–q9 minus q1) favor jsq most**, since jq and jaq
  must build and hold intermediate structures while jsq folds them away as it scans.
- **jsq's flat memory is a property of *reducing* queries, not projections.** Every query
  here except q1 collapses the input to a count / a handful of group rows, so jsq's owned
  memory stays at ~33 MiB no matter the file size. q1 is different: it emits one output
  row per surviving event (~3.5M rows at 1 GB, since ≈half clear `amount > 500`), and jsq
  buffers the entire result set on the heap before printing
  (`evaluator::run`, `engine/src/query/evaluator/mod.rs`). That buffer — a `path` string
  plus the projected value per row — is the high q1 figure, and it scales with the
  *output* size, not the file size. A query that returns a lot of rows costs real RAM on
  all three tools; jsq's advantage is on filters and aggregates that return little.
- **jq/jaq win at small scale.** On a few-MB file their lower startup makes them
  competitive or faster on the simple queries — the whole job is over before jsq's index
  build pays off. Reach for jsq when files are large; reach for jq/jaq for quick hits on
  small files and their far richer transformation language.
