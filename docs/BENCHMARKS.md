# Benchmarks: jsq vs jq vs jaq

`jsq`, [`jq`](https://jqlang.github.io/jq/), and [`jaq`](https://github.com/01mf02/jaq)
solve overlapping problems, so the fair question is: on the same file and the same query,
how do they compare on **wall time** and **memory**? `jaq` is a jq reimplementation in
Rust, which makes the trio especially informative — jq vs jaq isolates *C vs Rust*, while
jsq vs both isolates *parse-everything vs streaming*.

The short answer: jq and jaq both parse the whole document into the heap, so their memory
scales with file size. jaq is generally faster than jq (same approach, tighter
implementation), but it carries the same memory cost. `jsq` memory-maps the file and
streams a single pass, so its actual RAM stays nearly flat — tens of MB even on a 1 GB
file — and it pulls ahead on time as files grow. **The memory win is about the approach,
not the language: jsq beats jaq on RAM by the same wide margin it beats jq.**

The harness, generator, and queries are in [`../bench`](../bench) — everything here is
reproducible.

## Setup

- **Machine:** Apple M2 Max, 32 GB RAM, macOS (Darwin 25.2), SSD.
- **Tools:** `jsq` 1.0.0 (release build), `jq` 1.7.1, `jaq` 3.0.0, `hyperfine` 1.20.
- **Data:** a single JSON object `{"events":[ … ]}` — one large array of records with
  scalar fields plus a small nested `items` array (generated by
  [`bench/gen.py`](../bench/gen.py), seeded for reproducibility).
- **Time:** `hyperfine` with 1 warmup run, reporting the **median** of 5 runs (3 at
  1 GB). Output is redirected to `/dev/null` so streaming the result doesn't skew timing.

### Two memory numbers

Both come from `/usr/bin/time -l` on a separate single run:

- **RAM** = *peak memory footprint* (`phys_footprint`). This is the dirty + compressed
  memory the process actually owns — the number **Activity Monitor shows in its
  "Memory" column**. It excludes clean, file-backed pages the kernel can drop for free.
- **RSS** = *maximum resident set size*. This also counts the memory-mapped source and
  index pages faulted in during the scan. Those pages are clean and reclaimable (the
  kernel can evict them without swapping), so RSS overstates the memory jsq truly needs.

For `jq` and `jaq` the two are nearly identical — it's all heap. For `jsq` they diverge
sharply, and **RAM is the honest "how much memory does this need" figure.** RSS is
included for completeness.

## Queries

Four equivalent queries of increasing complexity. `jq` and `jaq` run the **same** filter
(jaq is jq-compatible); the jq form uses the efficient idiomatic shape — a `reduce` into
an object for group-bys, not `group_by` — so neither jq nor jaq is strawmanned. **All
three tools produce identical output, verified before timing.**

| #  | What it does                            | jsq                                                |
|----|-----------------------------------------|----------------------------------------------------|
| q1 | filter rows, project two fields         | `from .events[] as e where e.amount > 500 select { … }` |
| q2 | filter rows, count them                 | `… where e.status == "paid" aggregate { n: count() }`   |
| q3 | group by region, sum a field            | `aggregate { revenue: sum(e.amount) } by e.region`      |
| q4 | group by region, 3 metrics, sorted      | `aggregate { revenue, n, avg } by e.region order by …`  |

| q4 |   34 | 16967 | 15297 | 2352 | 12072 | 12573 |

On the aggregates jsq holds a flat **34 MiB** of real memory scanning the full gigabyte,
versus ~15 GB for jaq and ~17 GB for jq — a **~450–500× gap against both**. jaq is
~1.5–2× faster than jq here but needs essentially the same memory, since it too parses
the document into the heap. Both jq numbers also cross over (RAM > RSS) because macOS
compresses pages under memory pressure — `phys_footprint` counts the compressed copies
while RSS only counts what's still resident uncompressed.

## Takeaways

- **Memory is the headline, and it's an architecture story.** jq and jaq both hold the
  entire parsed document on the heap — roughly **16× the file size** — so a 1 GB file
  needs many GB of RAM whichever you pick. `jsq` keeps its records in a memory-mapped
  index, not the heap, so its actual RAM (the Activity-Monitor figure) stays in the tens
  of MB regardless of file size. At 100 MB that's ~33 MiB vs ~1.5–1.7 GB — and rewriting
  jq in Rust (jaq) barely moves that number, because the cost is the *approach*, not the
  language.
- **jaq is the speed control, not a memory control.** jaq is usually faster than jq
  (often ~1.5–2× on the aggregates) — a fair measure of what a tight Rust implementation
  of the parse-everything model buys you. It does not close the memory gap with jsq at
  all, which is the cleanest evidence that streaming, not Rust, is what makes jsq frugal.
- **mmap is why RSS and RAM differ for jsq.** jsq's larger RSS is mostly clean,
  file-backed pages (the source and the index) that the kernel can reclaim without
  swapping. That's also what lets jsq query files **larger than RAM** — something the
  parse-everything approach can't do at all.
- **Group-bys (q3/q4) favor jsq most**, since jq and jaq must build and hold intermediate
  structures while jsq folds them away as it scans.
- **jsq's flat memory is a property of *reducing* queries, not projections.** q2–q4
  collapse the input to a single count / a handful of group rows, so jsq's owned memory
  stays at ~34 MiB no matter the file size. q1 is different: it emits one output row per
  surviving event (~3.5M rows at 1 GB, since ≈half clear `amount > 500`), and jsq buffers
  the entire result set on the heap before printing
  (`evaluator::run`, `engine/src/query/evaluator/mod.rs`). That buffer — a `path` string
  plus the projected value per row — is the high q1 figure, and it scales with the
  *output* size, not the file size. A query that returns a lot of rows costs real RAM on
  all three tools; jsq's advantage is on filters and aggregates that return little.
- **jq/jaq win at small scale.** On a few-MB file their lower startup makes them
  competitive or faster on the simple queries — the whole job is over before jsq's index
  build pays off. Reach for jsq when files are large; reach for jq/jaq for quick hits on
  small files and their far richer transformation language.
